### 3. 命名规范

| 类型 | 规则 | 示例 |
|------|------|------|
| 文件名 | kebab-case | `pm-review-agent.ts` |
| 类型/接口 | PascalCase | `ReviewSession`, `ModuleInfo` |
| 函数 | camelCase | `reviewResume()`, `parseStructure()` |
| 常量 | UPPER_SNAKE_CASE | `PM_REVIEW_MODULES`, `MAX_RETRY` |
| 类型枚举数组 | UPPER_SNAKE + `as const` | `const PM_REVIEW_MODULES = [...] as const` |
| 环境变量 | UPPER_SNAKE_CASE | `SEARCH_API_KEY` |

### 4. 类型安全要求

- **所有函数参数和返回值必须有明确类型**，不允许 `any`
- **zod schema 与 TypeScript interface 保持同步**：`pm-review-parser.ts` 中的 zod schema 必须与 `types.ts` 中的接口一一对应
- **类型从 `@/lib/types` 统一导出**，不在各模块内部重复定义
- **LLM 返回内容一律经过 parser → guard 两层校验后才能使用**

### 5. LLM 调用规范

所有 LLM 调用必须通过 `@/lib/openai.ts` 中的 `callLLMWithMeta()` 或 `callLLM()`。

```typescript
// ✅ 正确：使用统一入口
const result = await callLLMWithMeta(systemPrompt, userPrompt, {
  maxTokens: 4096,
  responseFormat: "json_object",
  temperature: 0.3,
});

// ❌ 禁止：直接实例化 OpenAI client
const client = new OpenAI({ apiKey: "..." });
```

**参数约定**：
- `temperature`: 批阅类固定 `0.3`，创意类可适当提高
- `responseFormat`: 需要 JSON 输出时必须设置 `"json_object"`
- `maxTokens`: 按模块调整，结构识别 `2048`，项目经历批阅 `4096`，其他模块 `3072`

### 6. 错误处理规范

**分层错误处理**：

```
LLM 调用层
  → 网络/超时错误 → 记日志 + 返回友好错误
  → 输出为空/截断 → 记日志 + 触发重试

Parser 层
  → JSON 解析失败 → 抛出 PmReviewParseError（含 details）
  → 字段缺失/类型错误 → 抛出 PmReviewParseError

Guard 层
  → 校验失败 → 抛出 PmReviewGuardError（含 violations）

Agent 层
  → 单模块失败 → 记日志 + 跳过该模块，不阻塞整体
  → 搜索失败 → 降级为纯原文审计
  → 历史对比失败 → 跳过，返回无状态标记的结果

API 层
  → 统一捕获，返回 { success: false, error, details? }
```

**日志规范**：
```typescript
// 关键节点必须记录日志
console.log("[pm-review-agent] Step 2: 结构识别完成", { moduleCount: modules.length });
console.warn("[pm-review-agent] Step 4: 搜索工具调用失败，降级处理", { error: e.message });
console.error("[pm-review-agent] Step 3: 项目经历批阅失败", { error: e.message, attempt });
```

前缀统一为 `[pm-review-xxx]`，xxx 为模块名。

### 7. 重试策略

| 场景 | 最大重试 | 重试方式 |
|------|---------|---------|
| 单模块 LLM 调用 Parse 失败 | 1 次 | 拼接错误信息到 Prompt 重新调用 |
| 单模块 Guard 失败 | 1 次 | 拼接 violations 到 Prompt 重新调用 |
| 搜索工具调用失败 | 0 次 | 直接降级，不重试 |
| 结构识别失败 | 1 次 | 重试后仍失败则整体报错 |
| 历史对比失败 | 0 次 | 跳过，不阻塞 |

**重试时 Prompt 拼接方式**：沿用现有 `buildPmReviewParseRetryPrompt` / `buildPmReviewRetryPrompt` 模式。

### M5：Agent 编排引擎（★核心）

**开发思路**：
1. 新建 `pm-review-agent.ts`，作为整个批阅流程的调度中心
2. 主函数 `reviewResume(resumeText, sessionId?)` 按 7 步顺序执行
3. 每步独立 try-catch，单步失败有明确的降级策略
4. 支持进度回调（可选），便于后续接入前端进度展示

**核心流程伪代码**：

```typescript
export async function reviewResume(
  resumeText: string,
  sessionId?: string,
  onProgress?: (step: string, detail: string) => void,
): Promise<PmReviewComment[]> {
  
  // Step 1: 读取会话历史
  const previousReview = sessionId 
    ? await getLatestReview(sessionId) 
    : null;

  // Step 2: 结构识别
  const structure = await analyzeStructure(resumeText);
  //   → 输出 StructureAnalysis { modules[], missingModules[], redundantModules[] }
  //   → 失败：整体报错（这步是基础，不能跳过）

  // Step 3: 分模块批阅
  const allComments: PmReviewComment[] = [];
  for (const module of structure.modules) {
    try {
      const comments = await reviewModule(module, resumeText);
      allComments.push(...comments);
    } catch (e) {
      console.error(`[pm-review-agent] 模块 ${module.normalizedModule} 批阅失败`, e);
      // 跳过，继续下一个模块
    }
  }
  // 补充缺少模块的批注（add 类型）
  for (const missing of structure.missingModules) {
    allComments.push(buildMissingModuleComment(missing));
  }

  // Step 4: 数据验证
  const dataComments = allComments.filter(c => c.needsConfirmation && isDataIssue(c));
  if (dataComments.length > 0) {
    await enrichWithSearchEvidence(dataComments, resumeText);
  }

  // Step 5: 历史对比（仅多轮场景）
  if (previousReview) {
    await markPreviousRoundStatus(allComments, previousReview.comments);
  }

  // Step 6: Guard 校验
  const { comments: finalComments, dropped } = guardAndSanitize(resumeText, allComments);

  // Step 7: 保存
  if (sessionId) {
    await saveReviewResult(sessionId, resumeText, finalComments);
  }

  return finalComments;
}
```

**关键设计决策**：

| 决策点 | 方案 | 理由 |
|-------|------|------|
| 模块批阅顺序 | 按 structure.modules 顺序串行 | 串行可控、易调试；并行会增加复杂度且 API 有速率限制 |
| 结构识别失败 | 整体报错 | 后续步骤都依赖结构信息 |
| 单模块批阅失败 | 跳过该模块 | 不阻塞其他模块 |
| 搜索失败 | 保留原始批注不变 | searchEvidence 为空即可 |
| 历史对比失败 | 跳过，不标记状态 | previousRoundStatus 为空即可 |

**注意事项**：
- 每次 LLM 调用都要记录耗时和 token 消耗（console.log）
- `analyzeStructure` 返回的 `modules` 列表用于驱动后续循环，其结构必须稳定
- `reviewModule` 内部根据 `module.normalizedModule` 选择对应的 System Prompt
- Guard 校验复用现有 `sanitizePmReviewComments` + `validatePmReviewComments` 模式，但需升级
- 进度回调 `onProgress` 为可选参数，首版可以只打日志

```
POST /api/review-sessions
  → 请求 { studentName: string }
  → 返回 { session: ReviewSession }

GET  /api/review-sessions/[id]/history
  → 返回 { history: ReviewHistoryEntry[] }
```

**注意事项**：
- API 层只做参数校验和路由，业务逻辑全部在 lib 层
- 错误响应统一使用现有的 `jsonError` 模式
- 会话 API 不需要认证（个人工具）

## 第四部分：已确认的决策

| 决策项 | 结论 | 说明 |
|-------|------|------|
| 搜索服务 | 开发阶段 mock | 先用 mock provider 返回模拟数据，后续再接入真实搜索 API |

| 前端样式 | Tailwind CSS | 引入 Tailwind 替代 inline styles，提升开发效率和可维护性 |
| 进度展示 | 简单 loading | 不做 SSE 流式，批阅期间显示 loading 状态即可 |
