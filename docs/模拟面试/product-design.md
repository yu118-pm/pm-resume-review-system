# 模拟面试工具 — 产品设计文档

> 基于 [设计思路文档](./mock-interview-design-thinking.md) 补充界面结构与交互流程

---

## 一、设计目标

模拟面试工具的 V1 不是做“AI 面试官”，而是做一套顾问可直接使用的 **模拟面试准备工作台**。

因此界面设计必须服务三个目标：

1. **快速生成**：上传简历后，尽可能少的操作就能产出可用问卷
2. **快速判断**：先看到重点风险，再进入详细问题
3. **快速使用**：结果要按真实面试顺序组织，打开后就能直接拿来问

对应到产品形态，V1 最适合采用：

- **工作台式单页结构**
- **输入配置与结果浏览并存**
- **风险摘要 + 面试问卷的双层输出**

---

## 二、产品形态与信息架构

### 2.1 在整体工作台中的位置

该工具应作为“求职顾问工作台”中的一个独立工具，延续现有工作台结构：

- 左侧：工具导航
- 右侧：当前工具工作区

这样可以与“简历优化”等工具保持一致，避免每个工具都做成完全不同的页面范式。

### 2.2 模拟面试工具内部信息架构

模拟面试工具内部建议拆成四个层级：

1. **输入区**
2. **处理中状态区**
3. **结果概览区**
4. **详细问卷区**

信息优先级从上到下依次为：

```text
上传简历 / 粘贴文本
  ↓
生成模拟面试问卷
  ↓
重点风险提示（先看）
  ↓
候选人画像摘要
  ↓
按面试流程展开的详细问卷（主体）
```

原因是：

- 顾问真正最先需要的是“这份简历哪里最危险”
- 然后才是“按什么顺序去问”
- 不是一上来就看一大堆题目

## 三、页面布局设计

## 3.1 桌面端布局

V1 建议使用 **三栏逻辑、两层工作台**：

```text
┌──────────────┬──────────────────────┬────────────────────────────────────┐
│ 全局工具导航  │ 模拟面试配置面板       │ 结果工作区                           │
│              │                      │                                    │
│ 简历优化      │ 1. 上传/粘贴简历       │ A. 顶部结果操作条                    │
│ 模拟面试  ←   │ 2. 简历预览/修改       │ B. 风险提示卡                        │
│ 其他工具      │ 3. 生成按钮            │ C. 候选人画像摘要                    │
│              │ 4. 重新生成            │ D. 问卷模块导航                      │
│              │                      │ E. 问题卡列表（按流程）              │
└──────────────┴──────────────────────┴────────────────────────────────────┘
```

### 为什么不建议继续沿用“上输入、下结果”的单列结构

模拟面试的结果会明显比简历优化更长，尤其“项目深挖”模块容易占据大量屏幕空间。

如果输入区放在页面顶部，会有两个问题：

- 结果出来后，重新修改简历或重新生成需要频繁回到顶部
- 结果浏览时，输入区会占据过多首屏空间

因此更适合改为：

- **左侧固定配置面板**
- **右侧滚动结果区**

### 3.2 移动端布局

移动端退化为上下结构：

1. 顶部工具标题
2. 输入配置折叠面板
3. 风险提示
4. 问卷模块导航
5. 问卷详情

原则：

- 输入区默认折叠，避免挤压结果阅读
- 风险提示优先展示
- 模块导航采用横向滚动 chips

## 四、核心界面模块设计

## 4.1 左侧配置面板

该区域承担“输入、修正、生成、再生成”功能，信息密度要高，但交互要简单。

### 模块组成

#### 1. 工具说明卡

- 工具名：模拟面试
- 一句话说明：基于简历生成可直接使用的模拟面试问卷
- 版本标签：V1

#### 2. 简历输入区

支持两种方式：

- 上传 PDF / DOCX
- 直接粘贴简历文本

交互建议：

- 上传成功后自动解析文本
- 解析结果直接回填到文本框
- 提供“查看全文 / 编辑文本”入口
- 明确提示“你可先修正文档解析错误，再生成”

#### 3. 生成设置区

V1 不建议放太多参数，避免把简单工具做复杂。

建议只保留：

- 问卷题量：标准版（默认）/ 精简版
- 输出语言：中文（默认）

其中“标准版”默认 10-15 个主问题。

#### 4. 主行动区

主按钮：

- `生成模拟面试问卷`

辅助按钮：

- `重新生成`
- `清空当前内容`

#### 5. 处理中状态

点击生成后，在左侧面板展示分阶段进度，而不是只有一个转圈：

1. 解析简历结构
2. 识别风险点
3. 生成面试问卷

这样可以让等待过程更可理解，也更符合这个工具的内部处理逻辑。

---

## 4.2 右侧结果工作区

结果区需要先给判断，再给细节。

### 顶部结果操作条

建议包含：

- 生成时间
- `复制完整问卷`
- `导出 Markdown`
- `重新生成`

如果后续要扩展历史版本，可以在这里增加“切换最近一次结果”。

### 结果区结构

按照以下顺序展示：

1. 重点风险提示
2. 候选人画像摘要
3. 模块导航
4. 详细问卷

---

## 4.3 风险提示区

这是结果页最重要的首屏内容。

### 展示形式

建议采用 **3-5 张风险卡片**，每张卡只表达一个重点。

每张卡片包含：

- 风险标题
- 风险说明
- 风险等级：高 / 中
- 对应建议重点追问模块

示例：

- 项目成果表述偏弱
- 产品设计链路不完整
- 跨部门推动方式不明确
- 数据分析能力未被有效证明

### 关键交互

风险卡点击后，应直接联动到相关问题模块或高亮对应问题。

这样可以形成很强的使用闭环：

- 先看到风险
- 再跳到该风险对应的提问脚本

这会比单纯展示风险文字更实用。

---

## 4.4 候选人画像摘要区

风险下面需要有一个很轻的“候选人摘要”，帮助顾问快速进入语境。

建议包含 4 个摘要块：

- 求职方向
- 工作年限 / 经验阶段
- 主要经历结构
- 简历体现出的能力重心

注意：

- 这里只做摘要，不做长段解释
- 目标是帮助你快速理解“这是一个什么类型的候选人”

---

## 4.5 模块导航区

详细问卷很长，必须有快速定位能力。

建议使用横向模块导航条，固定在结果区顶部，按真实面试顺序排列：

1. 自我介绍
2. 常规人事 / 行政
3. 职业发展
4. 工作经历
5. 项目深挖
6. 产品思考
7. 技能素质
8. 数据分析 / 用户研究（按需出现）

### 导航交互

- 点击模块，滚动到对应模块
- 当前所在模块高亮
- 显示题量，例如 `项目深挖 5`

其中：

- **项目深挖模块默认展开**
- 其他模块可折叠展开

这样能把核心价值放到最前面。

---

## 4.6 问题卡设计

问卷主体不应该是一串纯文本，而应该是可快速浏览、可逐题使用的卡片结构。

### 每张问题卡建议包含

- 问题编号
- 主问题
- 观察点
- 回答思路
- 参考话术
- 可选追问

### 展示层级建议

默认展开：

- 主问题
- 观察点

折叠展开后显示：

- 回答思路
- 参考话术
- 可选追问

原因：

- 顾问在实际面试时最常看的首先是“问什么”和“观察什么”
- 参考话术和回答思路属于辅助信息，不应默认占满屏幕

### 单题交互建议

每道题提供轻量操作：

- `复制本题`
- `展开详情`
- `查看关联风险`

V1 不建议加入“记录回答”或“评分”入口，以免偏离产品定位。

## 五、视觉风格建议

## 5.1 整体风格

模拟面试工具建议延续当前工作台的专业基调，但比“简历优化”更强调判断感和结构感。

推荐方向：

- **Flat Design + Micro-interactions**
- **清晰层级 + 功能性色彩**
- **高可读性、弱装饰**

不建议：

- 做成聊天界面
- 做成 landing page 式大 Hero
- 用过强的渐变、拟物或花哨动效干扰阅读

### 视觉关键词

- 专业
- 稳定
- 有判断感
- 信息清晰

## 5.2 颜色建议

建议使用工作台风格的功能色体系：

- 主色：深蓝 / 墨蓝，用于主操作和导航高亮
- 背景：暖灰白或冷白，保证长时间阅读舒适
- 风险色：橙色 / 红色，用于风险提示与重点提醒
- 成功色：绿色，用于“已解析 / 已生成”

原则：

- 风险颜色只用于提醒，不大面积铺满
- 问卷区仍以中性色为主，避免视觉疲劳

## 5.3 动效建议

动效只用于加强反馈，不用于炫技。

适合加入的动效：

- 上传成功状态切换
- 生成过程分阶段 loading
- 风险卡 hover / focus 提示
- 模块导航滚动高亮
- 问题卡展开折叠

时长控制：

- 150ms - 250ms 为主

## 六、关键交互流程设计

## 6.1 主流程

```text
进入模拟面试工具
   ↓
上传简历 / 粘贴简历
   ↓
系统解析并回填文本
   ↓
用户检查并微调文本
   ↓
点击生成模拟面试问卷
   ↓
显示三阶段处理中状态
   ↓
生成完成，默认滚到结果区顶部
   ↓
先看风险提示
   ↓
通过模块导航查看详细问卷
   ↓
复制完整问卷或复制单题使用
```

---

## 6.2 上传与解析流程

### 交互目标

让用户明确感知：上传文件不是终点，**确认解析文本** 才是关键步骤。

### 流程设计

1. 用户上传 PDF / DOCX
2. 前端展示“正在解析简历”
3. 解析成功后：
   - 显示文件名
   - 显示成功提示
   - 自动打开或点开文本预览
4. 用户可直接修改解析后的文本
5. 若解析失败：
   - 明确错误原因
   - 提供“改为手动粘贴文本”

### 设计重点

- 不要只告诉用户“上传成功”，而要告诉用户“已解析，可继续检查文本”
- 解析失败时必须有兜底路径，不能卡死主流程

---

## 6.3 生成流程

### 点击生成后

按钮进入禁用态，并展示 loading 文案：

- `正在分析简历...`

同时左侧或结果区顶部展示阶段进度：

- 已完成：简历解析
- 当前进行中：风险识别
- 待完成：问卷生成

### 生成完成后

- 自动切换到结果状态
- 结果区滚动到顶部
- 默认聚焦“重点风险提示”

这一步非常重要，因为结果的第一阅读顺序应由产品引导，而不是让用户自己找重点。

---

## 6.4 结果浏览流程

### 浏览顺序设计

产品应主动引导用户按以下顺序浏览：

1. 风险提示
2. 候选人画像摘要
3. 项目深挖模块
4. 其他模块

这和模块自然顺序略有区别。

原因是：

- 风险决定了本次模拟面试的重点
- 项目深挖是 V1 的核心价值模块

因此可以在结果页中加入一个明显按钮：

- `优先查看项目深挖`

点击后直接跳到对应模块。

---

## 6.5 重新生成流程

重新生成是高频操作，设计上不能藏得太深。

建议支持两种方式：

1. 左侧配置区修改简历文本后直接重新生成
2. 结果区顶部操作条直接点击 `重新生成`

重新生成时保留：

- 当前简历文本
- 当前题量设置

不保留：

- 上一版高亮状态
- 临时展开状态

这样可以避免状态过多导致界面混乱。

---

## 七、状态设计

## 7.1 空状态

未上传简历前，右侧结果区不要留白。

应展示：

- 工具用途说明
- 生成后会得到什么
- 推荐使用步骤

例如：

- 先上传简历
- 检查解析文本
- 生成风险提示与模拟问卷

## 7.2 loading 状态

不只显示一个 spinner，建议使用：

- 阶段进度
- 骨架屏
- 预计输出内容占位

这样用户更容易理解“系统正在构建什么”。

## 7.3 异常状态

需要覆盖以下异常：

- 文件格式不支持
- 文件解析失败
- 简历内容过短
- 模型生成失败
- 结果部分缺失

异常提示必须尽量可操作，例如：

- `请改为粘贴纯文本简历`
- `请检查简历中是否包含完整项目经历`
- `本次未生成“数据分析”模块，因为简历中未体现相关经历`

## 7.4 低信息状态

如果简历信息非常薄弱，不应直接报错，而应降级输出：

- 风险提示照常给出
- 问卷数量减少
- 明确说明“部分问题因简历信息不足而无法深挖”

---

## 八、V1 页面方案总结

V1 最适合做成一个 **顾问工作台内的结构化结果工具**，而不是聊天型工具。

页面核心结构应为：

- 左侧固定配置面板
- 右侧结果工作区
- 结果先展示风险，再展示问卷
- 问卷按真实面试流程组织
- 项目深挖模块权重最高

这套方案的重点不是“看起来像 AI 产品”，而是：

- 让顾问在最短时间内找到最该问的内容
- 让结果足够有结构，能直接在模拟面试中使用
