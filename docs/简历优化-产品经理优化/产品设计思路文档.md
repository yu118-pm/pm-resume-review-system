## 一、产品定位

### 1.1 一句话定位

**用 AI 替代人工完成 PM 简历的专业批阅，输出带批注的 Word 文档，供学员参考修改。**

### 1.2 与现有工具的关系

```
简历优化工具（父级）
├── 通用简历优化（现有）
│   ├── 输入：简历 + 目标岗位 + 补充信息 + 经历类型
│   ├── 逻辑：AI 重写简历内容
│   └── 输出：优化后的新简历 + 优化说明
│
└── PM 简历批阅（新增）
    ├── 输入：仅简历文件
    ├── 逻辑：AI 按 PM 专业标准批阅
    └── 输出：带批注的 Word 文档
```

**设计原则**：两个工具独立并列，前端入口分开，技术底层复用。

## 二、核心设计思路

### 2.1 设计理念

**“AI Agent 扮演资深 PM 导师，分模块深度审查简历，结合外部信息验证，指出问题并给出修改建议，支持多轮追踪”**

关键点：
- **不是重写**：保留学员原文，只在有问题的地方插入批注
- **结构与内容并重**：既检查模块结构完整性，又评估内容质量
- **专业视角**：基于 PM 行业标准（STAR 法则、数据合理性、产品方法论等）
- **数据可验证**：通过搜索工具查询公司/行业信息，数据判断有据可依
- **多轮一致**：同一会话多次提交，前后批阅不矛盾
- **可操作性**：批注要具体，学员能直接参考执行
- **体验一致**：输出的 Word 批注与人工批阅体验一致

### 2.2 批阅逻辑流程（Agent 架构）

```
简历文件上传
  ↓
文件解析（PDF/DOCX → 文本）
  ↓
Agent 编排引擎启动
  ├── 读取会话历史（如有上次批阅结果）
  ├── 识别简历结构，拆分模块
  └── 制定批阅计划（哪些模块需要深入、哪些需要搜索验证）
  ↓
分模块批阅（每个模块独立调用 LLM）
  ├── 整体结构检查（是否缺少核心模块、是否有冗余模块）
  ├── 基础信息检查（必要字段、优势展示）
  ├── 自我评价检查（关键卖点提炼、是否空泛、是否结合经历）
  ├── 教育经历检查（课程简略化、核心成绩保留）
  ├── 工作经历检查（识别写法模式、按对应标准批注）
  ├── 项目经历检查（★核心：STAR、子结构完整性、数据合理性）
  │   └── 数据验证：Agent 自主决策是否调用搜索工具查询公司/行业信息
  └── 格式规范检查（加粗、错别字、模块顺序）
  ↓
合并所有模块批注结果
  ├── 如有历史批阅，对比前后一致性
  └── Guard 校验（anchorText 回溯、数据校验、一致性检查）
  ↓
前端预览 + 导出 Word 批注文档
  ↓
批阅结果持久化存储（供下次批阅参考）
```

### 2.3 批阅标准的分层设计

**第一层：各模块标准写法与检查方向（结构完整性 + 内容质量并重）**

| 模块 | 标准写法 | 检查方向 |
|------|----------|----------|
| 基础信息 | 必要部分（姓名、联系方式、经验）+ 优势部分（学历等） | 缺少必要字段、优势是否突出 |
| 自我评价 | 提炼关键卖点（行业/项目经验、核心能力） | 是否空泛、是否结合经历 |
| 教育经历 | 时间+学校+专业+学历，课程简略化 | 课程是否过多、核心成绩是否保留 |
| 工作经历 | 公司+岗位+时间（两种模式均可） | 识别写法模式，按对应标准批注 |
| 项目经历★ | 基本信息→背景→内容说明→职责(≤6条)→成果 | 结构完整性、描述质量、数据合理性 |
| 格式规范 | 标题加粗、模块顺序合理 | 错别字、冗余文字、排版 |
| 整体结构 | 包含所有核心模块 | 缺少模块、冗余模块、逻辑关系 |

**第二层：方向差异化标准**
- **传统 PM**：侧重需求分析、产品设计、跨团队协作
- **AI PM**：额外关注 AI 技术理解、模型效果、数据标注、算法协作

**第三层：数据合理性验证（Agent + 搜索工具）**
- 原文审计：口径、归因、自洽性
- 搜索验证：公司规模/行业基准（Agent 自主决策是否调用）

**实现方式**：AI 根据简历中的关键词（如“大模型”“推荐系统”“NLP”）自动判断方向，应用对应标准。

## 三、批阅输出结构设计

### 3.1 数据结构

每条批注的 JSON 结构：

```typescript
interface ReviewComment {
  sectionTitleOriginal: string;  // 原始区块标题
  normalizedModule?:             // 归类字段（可选）
    | "整体结构" | "基础信息" | "自我评价" 
    | "教育经历" | "工作经历" | "项目经历" | "格式";
  location: string;              // 定位描述
  anchorText: string;            // 原文摘录（用于 Word 批注定位）
  issueType: string;             // 问题类型
  actionType:                    // 建议动作类型
    | "rewrite" | "delete" | "merge" | "reorder" 
    | "format" | "verify" | "condense" | "add";
  comment: string;               // 批阅意见
  suggestion: string;            // 修改建议（必填）
  example?: string;              // 改写示例（可选）
  confidence: "high" | "medium" | "low";
  needsConfirmation: boolean;    // 是否需要确认真实性
  searchEvidence?: string;       // 搜索验证依据（如有）
  previousRoundStatus?: "new" | "modified" | "unchanged" | "resolved"; // 多轮状态
}
```

### 3.2 批注类型分类

| 模块 | 常见问题类型 |
|------|-------------|
| 整体结构 | 缺少核心模块、模块顺序不合理 |
| 自我评价 | 篇幅过长、空泛套话、缺乏方向感、与岗位脱节 |
| 项目经历 | 描述过于笼统、过于冗长、缺少成果、数据不合理、相似模块未合并、缺少收尾 |
| 工作经历 | 职责描述不体现 PM 角色、缺乏职业成长线 |
| 技能 | 篇幅过长、冗余表达、未分组 |
| 教育背景 | 课程列表过多、荣誉证书缺项目名 |
| 格式 | 关键标题未加粗、错别字 |

### 3.3 批注示例

**示例1：项目经历 - 描述过于笼统**
```json
{
  "module": "项目经历",
  "location": "CRM项目 - 第2条职责",
  "type": "描述过于笼统",
  "originalText": "负责订单管理功能的制作",
  "comment": "职责描述过于笼统，缺少具体做了什么功能、支持了什么场景",
  "suggestion": "主导订单管理板块，支持包括特卖、拼团和砍价等订单的管理，有效提供公司管理维护不同的订单"
}
```

**示例2：自我评价 - 篇幅过长**
```json
{
  "module": "自我评价",
  "location": "自我评价段落",
  "type": "篇幅过长",
  "originalText": "我是江西理工大学大四毕业生，深耕 B 端电商采购领域产品实习...",
  "comment": "信息太密了，可以简化凸显重要信息就行，比如：江西理工大学 2026 届本科生，具备企业采购平台产品实习经历，参与 B 端 aPaaS 电商采购平台的需求分析、原型设计、PRD 输出与跨团队协作...",
  "suggestion": null
}
```

**示例3：项目经历 - 数据不合理**
```json
{
  "module": "项目经历",
  "location": "商城模块 - 登录注册",
  "type": "数据不合理",
  "originalText": "新用户注册成功率从 72% 提升至 90%",
  "comment": "该结果缺少统计周期、样本范围和归因依据，当前无法验证该提升是否能直接归因到该模块设计。",
  "suggestion": "如果没有稳定统计口径，建议改为定性表达；如果有口径，可补充统计周期和归因依据后再保留数字。"
}
```

## 四、用户流程设计

### 4.1 首次批阅流程

```
1. 进入工具，选择"PM 简历批阅"
   ↓
2. 创建批阅会话（关联学员信息）
   ↓
3. 上传简历文件（PDF/DOCX）
   ↓
4. 点击"开始批阅"
   ↓
5. Agent 分模块批阅（显示进度：解析 → 结构识别 → 分模块批阅 → 数据验证 → 汇总）
   ↓
6. 预览批阅结果
   ├── 按模块分组展示所有批注
   ├── 每条批注显示：定位 + 原文 + 批阅意见 + 修改建议/示例
   ├── 搜索验证的批注显示验证依据
   └── 支持展开/折叠
   ↓
7. 导出并下载带批注的 Word 文档
   ↓
8. 将文档发给学员，学员根据批注修改简历
```

### 4.2 多轮批阅流程

```
1. 找到该学员的批阅会话
   ↓
2. 上传修改后的简历
   ↓
3. 点击"再次批阅"
   ↓
4. Agent 读取上次批阅结果 + 分模块批阅
   ├── 对比上次批注，识别已修改/未修改/新增内容
   ├── 已修改的重新评估
   ├── 未修改的保留提醒
   └── 新增内容做新批注
   ↓
5. 预览批阅结果（标注每条批注的多轮状态）
   ↓
6. 导出新的批注文档
```

### 4.3 异常流程

**文件解析失败**
- 提示：文件格式不支持或文件损坏，请上传 PDF 或 DOCX 文件
- 允许重新上传

**Agent 批阅失败**
- 提示：批阅失败，请重试
- 记录错误日志供排查（含各模块处理状态）

**搜索工具失败**
- 不影响整体批阅流程，数据验证部分降级为纯原文审计
- 批注中标注"未能获取外部验证信息"

**批注数量异常**
- 批注过少（<3 条）：提示"简历整体质量较好，仅有少量建议"

## 六、AI Prompt 设计思路（Agent 架构）

### 6.1 整体设计

Agent 架构下，Prompt 不再是单一的 System + User，而是分层设计：

```
Agent 编排层（Orchestrator）
  ├── 结构识别 Prompt：解析简历结构，拆分模块
  ├── 各模块批阅 Prompt：每个模块专属的批阅指令
  ├── 数据验证 Prompt：结合搜索结果判断数据合理性
  ├── 一致性检查 Prompt：对比上次批阅结果（多轮场景）
  └── 汇总 Prompt：合并各模块结果，去重、排序
```

### 6.2 各模块 Prompt 设计原则

- 每个模块的 Prompt 包含：角色定位 + 该模块的标准写法 + 检查方向 + 输出格式
- 不限批注数量，要求覆盖所有值得指出的问题
- 结构完整性与内容质量并重
- 无需修改的内容不标注

### 6.3 工具定义（Agent 可调用）

| 工具 | 功能 | 调用时机 |
|------|------|----------|
| search_company | 搜索公司基本信息（规模、行业、主营业务） | 项目经历中涉及数据验证时 |
| search_industry | 搜索行业基准数据（转化率、留存率等） | 数据量级可疑时 |
| get_review_history | 读取上次批阅结果 | 多轮批阅场景 |
| save_review_result | 保存本次批阅结果 | 批阅完成后 |

详细 Prompt 设计见《AI 处理与 Prompt 设计》文档。

## 七、Word 批注生成方案

### 7.1 技术方案

使用 **docx-js** 库（与现有导出简历 Word 的技术栈一致）实现 Word 批注写入。

### 7.2 批注插入逻辑

```
1. 读取原始简历文件（DOCX）
   ↓
2. 解析文档结构（段落、文本块）
   ↓
3. 遍历 AI 批阅结果
   ↓
4. 对每条批注：
   ├── 根据 anchorText 在文档中定位对应段落/文本
   ├── 在该位置插入 Word Comment
   └── Comment 内容 = comment + suggestion + searchEvidence(如有)
   ↓
5. 保存为新的 DOCX 文件
   ↓
6. 返回给前端下载
```

### 7.3 批注内容格式

Word 批注气泡中的内容格式：

```
【问题】描述过于笼统

职责描述过于笼统，缺少具体做了什么功能、支持了什么场景。

【建议】
主导订单管理板块，支持包括特卖、拼团和砍价等订单的管理，有效提供公司管理维护不同的订单
```

### 7.4 定位策略

**优先级**：
1. 精确匹配 anchorText（最准确）
2. 模糊匹配（去除空格、标点后匹配）
3. 关键词匹配（提取核心词匹配）

**定位失败处理**：
- 如果无法定位，将批注插入到对应模块的开头
- 在批注中说明"未能精确定位，请参考原文位置"

```
POST /api/review-pm-resume { resumeText, sessionId }
  → Agent 编排引擎启动
  → 读取会话历史（如有）
  → 结构识别（LLM）
  → 分模块批阅（多次 LLM 调用）
  → 搜索验证（Agent 自主决策）
  → 历史对比（多轮场景）
  → Guard 校验 + 汇总
  → 保存批阅结果到会话历史
  → 返回 { success: true, comments: [...] }
  ↓
前端展示批阅结果（含多轮状态标注）
  ↓
用户点击"导出 Word"
  ↓
POST /api/export-review-docx
  → 接收原始文件 + comments
  → insertCommentsToDocx(file, comments)
  → 返回带批注的 DOCX 文件流
  ↓
前端触发下载
```

## 九、关键技术点

### 9.1 AI 批阅质量保障

**策略1：分模块 Prompt 工程**
- 每个模块独立 Prompt，包含该模块的标准写法和检查方向
- 提供正面和反面示例
- 强调"无需修改的内容不标注"
- 不限批注数量，要求覆盖所有值得指出的问题

**策略2：温度控制**
- 使用较低的 temperature（0.3）保证稳定性

**策略3：Guard 校验**
- 检查 anchorText 是否在简历中存在（回溯验证）
- 检查批注是否有明显的误判（如批注内容与原文无关）
- 多轮场景下检查前后批注一致性

### 9.2 Word 批注定位准确性

**策略1：文本预处理**
- 统一换行符、空格
- 去除不可见字符

**策略2：多级匹配**
- 优先精确匹配
- 降级到模糊匹配
- 最后用关键词匹配

**策略3：定位验证**
- 前端预览时显示批注定位位置
- 允许用户反馈定位错误

### 9.3 数据合理性判断（Agent + 搜索工具）

**第一层：原文审计**（纯 LLM）：

```
1. 检查是否具备统计口径
   - 是否有时间窗、样本范围、分母或基线

2. 检查是否具备归因依据
   - 是否能说明结果与候选人的动作直接相关

3. 检查是否前后自洽
   - 同一项目内数据和表述是否冲突
```

**第二层：搜索验证**（Agent 自主决策是否调用）：

```
1. 搜索公司基本信息
   - 规模、行业、主营业务

2. 搜索行业基准数据
   - 如电商平均转化率、SaaS 行业留存率等

3. 结合搜索结果判断数据量级是否匹配
```

**批注规则**：
- 口径缺失 → 建议补充统计依据
- 无法验证 → 建议改为定性描述
- 量级可疑 → 提醒需确认，不直接改写新数字
- 搜索信息仅作参考，不确定时标注"建议确认"

### 9.4 多轮批阅一致性

**会话存储方案**：
- 每个学员对应一个批阅会话（sessionId）
- 每次批阅结果保存为会话历史记录（含时间戳、简历文本、批注数组）
- 再次批阅时，Agent 读取上次批阅结果作为上下文

**对比逻辑**：
- 已修改的内容：重新评估，标记为 `modified`
- 未修改的内容：保留提醒，标记为 `unchanged`
- 新增内容：做新批注，标记为 `new`
- 上次问题已解决：标记为 `resolved`

## 十二、风险与应对（补充）

### 风险4：批注内容与原文不符

**表现**：批注指出的问题在原文中并不存在

**应对**：
- Prompt 中强调"必须基于原文批阅，不能编造问题"
- Guard 校验：检查 anchorText 是否在简历中存在
- 前端预览时允许用户标记"误判"，收集反馈优化 Prompt

### 风险5：批注建议不专业

**表现**：修改建议不符合 PM 行业标准

**应对**：
- Prompt 中内置大量正面示例
- 定期用真实简历测试，收集宇老师反馈
- 迭代优化批阅标准和 Prompt

### 风险6：多轮批阅前后矛盾

**表现**：学员修改后再次提交，AI 给出与上次矛盾的批注

**应对**：
- Agent 必须读取上次批阅结果作为上下文
- 一致性检查 Prompt 专门对比前后批注
- Guard 校验检查逻辑冲突

### 风险7：搜索工具返回信息不准确

**表现**：搜索到的公司/行业信息与实际不符

**应对**：
- 搜索信息仅作参考，不作为确定性结论
- 信息不确定时批注中标注"建议确认"
- 搜索失败时降级为纯原文审计
