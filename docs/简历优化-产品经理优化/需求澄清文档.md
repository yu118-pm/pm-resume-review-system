## 一、需求背景

### 1.1 业务场景

宇老师负责产品经理培训业务，学员完成课程学习后，需要根据自己的工作经历和项目经历撰写简历。当前流程是：

1. 学员自行撰写简历
2. 提交给宇老师人工批阅
3. 宇老师在 Word 文档中用批注功能标注问题和修改建议
4. 学员收到批阅后的简历，自行参考修改

### 1.2 核心痛点

- **批阅工作量大**：每份简历需要逐模块、逐条审查，耗时较长
- **批阅标准难统一**：人工批阅时难以保证每份简历都覆盖所有检查点
- **批量处理困难**：一期学员可能有 20+ 份简历，人工批阅压力大
- **批阅质量依赖状态**：疲劳时可能遗漏问题或批注不够详细

### 1.3 现有工具的局限

项目中已有"通用简历优化"工具，但其核心逻辑是**重写简历内容**，输出优化后的新简历。这与 PM 简历批阅的需求不符：

| 维度 | 通用简历优化 | PM 简历批阅需求 |
|------|-------------|----------------|
| 目标 | 生成更好的简历 | 指出问题，指导学员自行修改 |
| 输出 | 全新的简历内容 | 带批注的原简历 |
| 适用对象 | 任何求职者 | PM 培训学员 |
| 专业深度 | 通用优化 | PM 专业视角的深度批阅 |

因此需要独立设计一个新工具。

### 1.4 单次 LLM 调用的局限

初版设计中，AI 批阅采用单次 LLM 调用完成所有模块的批阅。在实际测试中发现以下局限：

| 问题 | 说明 |
|------|------|
| 批注数量受限 | 单次调用输出长度有限，难以覆盖全部问题 |
| 数据验证不足 | 纯 LLM 无法查询公司/行业信息，数据合理性判断缺乏依据 |
| 多轮一致性缺失 | 学员修改后再次提交时，无法参考上次批阅结果，可能前后矛盾 |
| 模块深度不均 | 一次性批阅所有模块，项目经历等重点模块的批阅深度不够 |

因此 V2 设计升级为 **Agent 架构**，具备分模块批阅、工具调用（搜索验证）、历史上下文等能力。

## 二、需求目标

### 2.1 核心目标

**用 AI 替代人工完成 PM 简历的专业批阅，输出带批注的 Word 文档，供学员参考修改。**

### 2.2 成功标准

- **批阅质量**：AI 批阅结果与宇老师人工批阅的标准基本一致
- **批阅覆盖**：全面覆盖简历的所有模块，不限批注数量，目标是简历完全批注
- **结构与内容并重**：既检查模块结构完整性，又评估内容质量
- **数据可信度**：通过搜索工具验证公司/行业信息，数据合理性判断有据可依
- **多轮一致性**：学员修改后再次提交时，批阅结果与上次不矛盾
- **输出可用性**：导出的 Word 文档中的批注清晰、定位准确、建议可操作
- **效率提升**：单份简历批阅时间从人工 30-60 分钟降至 AI 处理数分钟
- **可扩展性**：验证质量后，能支持批量处理（一期 20+ 份简历）

## 三、核心问题与解决方向

### 3.1 问题1：如何定义 PM 简历的批阅标准？

**问题描述**：  
不同于通用简历优化，PM 简历有其专业特性（如 STAR 法则、产品方法论、数据合理性等）。需要明确的批阅标准才能指导 AI。

**解决方向**：  
基于宇老师的人工批阅经验和简历写作方法论，梳理出以下批阅框架（结构完整性 + 内容质量并重）：

#### 批阅框架：各模块标准写法与检查方向

**1. 基础信息**
- 必要部分：姓名、联系方式、经验
- 优势部分：如学历（好学历应突出展示，否则正常标注即可）
- 检查方向：是否缺少必要字段、优势是否得到合理展示

**2. 自我评价 / 个人技能 / 优势**
- 核心要求：提炼关键卖点（行业经验、项目经验、核心能力），不要空泛表达
- 检查方向：是否空洞套话、是否缺乏方向感、是否结合了工作经验和项目经历

**3. 教育经历**
- 标准内容：时间、学校、专业、学历
- 课程：如有则简略化展示核心课程即可
- 成绩/荣誉：如有突出成绩（比赛、奖项等）保留核心重要的即可

**4. 工作经历**
- 标准内容：任职公司、岗位、时间
- 两种写法均可接受：
  - ① 独立板块：公司+岗位+时间 + 该公司的核心工作内容、价值
  - ② 非独立：仅公司+岗位+时间，项目经历直接挂在各公司下方
- 检查方向：AI 先识别学员采用的是哪种模式，再按对应模式的完整性要求批注

**5. 项目经历**（★ 核心模块）
- 总体原则：STAR 法则
- 子结构检查：
  - 基本信息：项目名称、岗位
  - 项目背景/简介：简要介绍当前项目是什么
  - 内容说明：自己负责做的具体相关事项，项目中什么功能模块是自己做的
  - 项目职责：一般不超过 6 条
  - 项目成果：取得了什么成绩
- 职责与成果的写法：可以分开写，也可以整合为一个板块，两种写法都接受，内容质量到位即可
- 检查方向：结构是否完整（是否缺背景、缺成果）、描述是否笼统、数据是否合理、篇幅是否合理

**6. 格式规范**
- 关键标题是否加粗
- 模块顺序是否合理
- 是否存在错别字或冗余文字

**7. 整体结构**
- 是否缺少核心模块（如完全没有项目经历）
- 模块间的逻辑关系是否合理
- 是否有无价值的冗余模块

**已确认的标准**：见附录《PM 简历批阅标准框架》

### 3.2 问题2：如何判断简历中的数据是否合理？

**问题描述**：  
学员简历中常出现数据虚高或不可信的情况（如实习生写"提升 90%"），但如何修改才合理，宇老师在人工批阅时也难以把握。

**解决方向**：  
升级为 **Agent + 搜索工具** 方案，AI 不仅做原文审计，还能调用搜索工具获取外部信息辅助判断：

1. **原文审计（基础层）**  
   - 检查是否有统计口径（时间窗、样本范围、基线）
   - 检查归因依据是否充分
   - 检查前后数据是否自洽

2. **搜索验证（增强层，Agent 自主决策是否调用）**  
   - 搜索公司基本信息：规模、行业、主营业务
   - 搜索行业基准数据：如电商平均转化率、SaaS 行业留存率等
   - 结合搜索结果判断数据量级是否与公司规模/行业特征匹配

3. **批注原则（不变）**  
   - 不得把原文数字改写成另一个“更合理”的数字
   - 口径缺失 / 无法验证 → 建议补充统计依据
   - 数据风险较高 → 建议改为定性描述
   - 数据合理 → 不标注
   - 没有数据时可用定性成果替代

### 3.3 问题3：批阅输出的形式是什么？

**问题描述**：  
AI 批阅后如何呈现给学员？是纯文本列表，还是直接在 Word 中标注？

**解决方向**：  
**使用 Word 原生批注（Comment）功能**，在原简历对应位置插入批注气泡，内容包括：
- 问题说明
- 修改建议
- 修改示例（可选，不是每条都给）

**优势**：
- 学员打开 Word 就能看到侧边批注，体验与人工批阅一致
- 批注定位精准，学员知道具体哪里有问题
- 无需修改的内容保持原样，不干扰阅读

---

### 3.4 问题4：批阅的粒度和力度如何控制？

**问题描述**：  
是每个模块都批，还是只批重点模块？每条批注的详细程度如何？

**已确认的要求**：
- **批阅范围**：全部简历内容，分模块多次调用 LLM，确保每个模块批阅深度充分
- **批注数量**：不限上限，目标是简历完全批注，所有值得指出的问题都要覆盖
- **批注原则**：无需修改的内容不标注，有问题的内容才批注
- **批注详细度**：指出问题 + 给出修改方向或示例（参考宇老师的批阅案例风格）

---

### 3.5 问题5：如何区分传统 PM 和 AI PM？

**问题描述**：  
学员方向包括传统产品经理和 AI 产品经理，批阅标准是否有差异？

**解决方向**：  
- **共性标准**：STAR 法则、数据合理性、篇幅控制等适用于所有 PM
- **差异化标准**：
  - **传统 PM**：侧重需求分析、产品设计、跨团队协作、业务理解
  - **AI PM**：额外关注是否体现了对 AI 技术的理解、模型效果评估、数据标注管理、AI+业务场景落地能力、与算法团队的协作方式

**实现方式**：  
AI 根据简历中体现的项目类型（如出现“大模型”“推荐系统”“NLP”等关键词）自动判断方向，应用对应的差异化标准。

---

### 3.6 问题6：如何支持多轮批阅？

**问题描述**：  
学员修改简历后再次提交批阅，AI 需要知道之前的批阅情况，避免前后两次批阅自相矛盾。

**解决方向**：  
采用 **会话制** 设计：
- 每个学员对应一个批阅会话，导师在同一会话中多次提交修改版本
- 每次批阅结果持久化存储，作为下次批阅的上下文输入
- Agent 在批阅时对比上次批注，判断哪些已修改、哪些未改、新增/调整的内容做新批注
- 确保前后批阅结果不矛盾

---

### 3.7 问题7：为什么需要 Agent 架构？

**问题描述**：  
单次 LLM 调用无法同时满足“分模块深度批阅 + 搜索验证 + 历史一致性”的要求。

**解决方向**：  
采用 Agent 架构，具备以下能力：

| 能力 | 说明 |
|------|------|
| 分模块批阅 | 每个模块独立调用 LLM，确保批阅深度 |
| 搜索验证 | 调用搜索工具查询公司/行业信息，辅助数据合理性判断 |
| 历史对比 | 读取上次批阅结果，避免前后矛盾 |
| 自主决策 | Agent 自行决定是否需要搜索、是否需要深入某个模块 |

## 四、需求边界

### 4.1 本期做什么

✅ **核心功能**：
- 单份简历上传 → Agent 批阅 → 导出带批注的 Word 文档
- 分模块深度批阅，不限批注数量，目标完全批注
- 前端预览批阅结果
- 基于 PM 专业标准的深度批阅（结构完整性 + 内容质量并重）
- 多轮批阅：同一会话中多次提交修改版本，保持前后一致性

✅ **Agent 能力**：
- 分模块独立调用 LLM，确保每个模块批阅深度
- 搜索工具查询公司/行业信息，辅助数据合理性判断
- 读取上次批阅结果，避免前后矛盾
- 自主决策是否需要搜索、是否需要深入某个模块

✅ **技术实现**：
- 复用现有文件解析能力（PDF/DOCX → 文本）
- Agent 编排引擎 + 分模块 Prompt
- 搜索工具集成（公司信息 + 行业基准）
- 会话历史存储与对比逻辑
- Word 批注（Comment）写入能力

### 4.2 本期不做什么

❌ **批量处理**：先验证单份批阅质量，后续再加批量模式  
❌ **在线编辑**：不提供在线修改简历的能力，只导出批注文档  
❌ **评分机制**：不对简历打分，只给批注建议  
❌ **深度数据验证**：不尝试验证具体项目是否真实存在，只做中等深度验证

### 4.3 与现有工具的关系

- **独立并列**：PM 简历批阅与通用简历优化是两个独立的工具，前端入口分开
- **技术复用**：文件解析、LLM 调用等基础能力复用
- **Prompt 独立**：批阅 Prompt 与优化 Prompt 完全独立，互不影响

## 五、关键约束与要求

### 5.1 批阅质量要求

- **准确性**：批注的问题必须真实存在，不能误判
- **专业性**：批注需体现 PM 专业视角，不是通用建议
- **可操作性**：修改建议要具体，学员能直接参考执行
- **完整性**：不遗漏明显问题

### 5.2 输出格式要求

- **Word 批注格式**：使用 Word 原生 Comment 功能
- **批注定位**：精准定位到原文对应位置（段落级或句子级）
- **批注内容结构**：问题说明 + 修改建议/示例
- **保留原格式**：导出的 Word 文档保持原简历的排版和格式

### 5.3 性能要求

- 单份简历批阅时间：数分钟内（含多模块 Agent 处理 + 搜索验证 + Word 生成）
- 支持的简历文件格式：PDF、DOCX
- 文件大小限制：≤ 10MB

## 六、典型使用场景

### 场景1：首次提交简历批阅

1. 导师为学员创建批阅会话
2. 上传学员简历文件（PDF/DOCX）
3. 点击"开始批阅"
4. Agent 分模块批阅（可能调用搜索工具验证数据）
5. 预览批阅结果（按模块分组展示所有批注）
6. 下载带批注的 Word 文档
7. 将文档发给学员，学员根据批注修改简历

### 场景2：学员修改后再次提交（多轮批阅）

1. 导师找到该学员的批阅会话
2. 上传修改后的简历
3. Agent 读取上次批阅结果，对比修改情况
4. 对已修改的内容重新评估，对未修改的提醒，对新增内容做新批注
5. 确保前后批阅不矛盾
6. 预览并导出新的批注文档

### 场景3：导师验证批阅质量

1. 上传一份已人工批阅过的简历
2. 对比 AI 批阅结果与人工批阅结果
3. 检查是否有遗漏或误判
4. 根据对比结果调整批阅标准或 Prompt

## 七、风险与应对

### 风险1：AI 批阅质量不稳定

**表现**：同一份简历多次批阅结果差异大，或批注质量时好时坏

**应对**：
- Prompt 中明确批阅标准和输出格式要求
- 使用较低的 temperature（如 0.3）保证稳定性
- 增加 Guard 校验，过滤明显不合理的批注

### 风险2：Word 批注定位不准确

**表现**：批注位置与问题内容对不上

**应对**：
- AI 输出时要求明确标注原文片段
- Word 批注插入时基于文本匹配定位
- 前端预览时让用户确认批注位置是否准确

### 风险3：批注过少

**表现**：一份简历只有 2-3 条批注，明显覆盖不全

**应对**：
- 分模块批阅确保每个模块都被审查
- 批注数量过少时前端提示"简历整体质量较好，仅有少量建议"

### 风险4：多轮批阅前后矛盾

**表现**：学员修改后再次提交，AI 给出与上次矛盾的批注

**应对**：
- Agent 必须读取上次批阅结果作为上下文
- 对已修改内容重新评估时，明确参考上次的修改建议
- Guard 校验检查前后批注是否存在逻辑冲突

### 风险5：搜索工具返回信息不准确

**表现**：搜索到的公司/行业信息与实际不符，导致数据判断偏差

**应对**：
- Agent 对搜索结果标注可信度
- 搜索信息仅作参考，不作为确定性结论
- 信息不确定时批注中标注"建议确认"
