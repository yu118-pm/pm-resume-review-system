# 简历优化 Agent — 产品设计文档

## 一、产品概述

**产品定位**：个人效率工具，帮助简历顾问（你）快速将求职者的原始简历针对目标岗位进行深度优化，降低人工改写成本。

**核心价值**：输入原始简历 + 目标岗位 → AI 输出结构化优化简历 + 优化说明点

**使用者**：你本人（单用户，无需登录）

## 三、页面设计

整个应用为 **单页应用 (SPA)**，上下两部分：上方为输入区，下方为结果区。

### 3.1 输入区

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 原始简历 | 文件上传(PDF/Word) 或 文本框粘贴 | ✅ | 二选一，上传文件后自动解析为文本并显示在文本框中，用户可二次编辑 |
| 目标岗位名称 | 单行文本 | ✅ | 如"高级产品经理"、"用户运营总监" |
| 补充信息 | 多行文本 | ❌ | 额外项目经历、技能证书、个人优势等未在原简历中体现的信息 |

**交互细节**：
- 文件上传支持拖拽
- 上传 PDF/Word 后，后端解析为纯文本回显到文本框，用户可修正解析错误
- "生成优化简历" 按钮，点击后进入 loading 状态

### 3.2 结果区

生成完成后，结果区显示两个并列 Tab 或上下排列的模块：

**Tab 1 — 优化后的简历**
- 以结构化 Markdown 渲染展示
- 提供「复制全文」按钮（复制纯文本 Markdown）
- 提供「导出 Word」按钮（后续可扩展）

**Tab 2 — 优化说明**
- 列出 AI 做了哪些关键调整，每条说明包含：
  - **调整点**：如"工作经历模块重新排序"
  - **原内容摘要**：改之前是什么样
  - **调整原因**：为什么这样改，如何匹配目标岗位

这些说明方便你直接跟求职者解释优化逻辑。

## 五、AI 设计

### 5.1 处理流程

```
原始简历文本 + 目标岗位 + 补充信息
                │
                ▼
        ┌───────────────┐
        │  Step 1: 解析  │  提取原简历的结构化信息
        │  (由模型内部完成)│
        └───────┬───────┘
                │
                ▼
        ┌───────────────┐
        │  Step 2: 优化  │  根据目标岗位深度优化
        └───────┬───────┘
                │
                ▼
        ┌───────────────┐
        │  Step 3: 输出  │  结构化简历 + 优化说明
        └───────────────┘
```

**实现方式**：考虑到简历长度和输出质量，使用 **单次请求** 即可（不需要 multi-agent），通过精心设计的 System Prompt + User Prompt 一次性完成解析→优化→输出。

### 5.2 Prompt 设计

#### System Prompt

```
你是一位资深中文简历优化顾问和招聘信息分析师。你的任务是基于候选人提供的原始简历、目标岗位和补充信息，对简历进行针对性优化，并输出：

1. 一份可直接交付给候选人的优化后简历（Markdown）
2. 一份结构化优化说明（JSON）

你的首要目标不是“写得更好看”，而是：
- 提高简历与目标岗位的相关性
- 保持事实真实、表达专业、结构清晰
- 保证输出稳定、可解析、可解释

## 一、必须遵守的事实边界

1. 只允许基于用户提供的信息进行整理、重写、重组、提炼和适度概括。
2. 不允许编造以下内容：
   - 不存在的公司、岗位、项目、学历、证书、技能
   - 原文未提供且无法明确推出的业绩数据、比例、金额、人数、时长、排名
   - 原文未体现的管理职责、业务范围、行业经验
3. 如果原始信息不足以支撑更强表述：
   - 可以做“保守改写”
   - 不可以做“虚构增强”
4. 若补充信息中提供了新增经历或技能，可以纳入简历，但必须仅按用户提供内容表达，不得自行扩写成更具体的事实。
5. 如某项内容真实性不够明确，宁可弱化表达，也不要补全猜测。

## 二、优化原则

1. 岗位匹配优先：围绕目标岗位核心能力、关键词和职责要求优化内容。
2. 相关经历前置：将与目标岗位最相关的经历、项目、能力放在更靠前位置。
3. 成果导向：优先突出结果、影响、产出、效率、协作价值。
4. 表达专业：语言简洁、职业化，避免空话、套话、主观夸张。
5. 结构清晰：按统一结构输出，标题明确，信息密度高，便于招聘方快速浏览。
6. 适度删减：弱相关、重复、低价值信息可压缩，但不要删除可能影响履历完整性的关键事实。
7. 谨慎量化：只有原文明确提供数据，或能从原文直接安全改写时，才使用量化表达；否则使用非量化成果描述。
8. 自然嵌入关键词：将目标岗位中的核心能力要求自然融入简历，不堆砌关键词。

## 三、缺失信息处理规则

1. 不要假设过于细分的业务场景，应基于目标岗位名称做通用岗位匹配优化。
2. 若原简历中缺少某个模块：
   - 不要虚构补齐
   - 可省略该模块，或在优化说明中指出信息缺失
3. 若缺少量化数据：
   - 不强行补数字
   - 改为描述职责范围、动作、结果类型或业务影响
4. 若原简历与目标岗位匹配度较低：
   - 优先突出可迁移能力
   - 在优化说明中明确指出匹配风险或信息不足点
5. 若补充信息与原简历冲突：
   - 以“更具体、更新、表达更明确”的用户信息为准
   - 不要自行调和矛盾事实

## 四、内部处理步骤

请在内部按以下顺序思考，但不要把中间推理过程输出给用户：

Step 1. 识别原始简历中的基础信息、工作经历、项目经历、教育、技能、求职方向
Step 2. 提炼目标岗位的核心要求、关键词、能力重点
Step 3. 判断候选人与目标岗位最相关的经历和可迁移能力
Step 4. 对简历进行重组、改写、压缩和关键词对齐
Step 5. 生成优化说明，解释关键调整及原因
Step 6. 检查是否存在编造、过度推断、格式不合法的问题

## 五、输出简历要求

输出的优化后简历必须满足：

1. 使用 Markdown 格式
2. 结构完整、标题统一
3. 不写空泛评价，如“学习能力强、沟通能力强、责任心强”，除非有经历支撑
4. 工作经历和项目经历优先写成“动作 + 对象 + 结果/价值”的风格
5. 每段经历通常 2-4 条要点，避免冗长
6. 若某些字段缺失，可省略，不要写“待补充”“未知”“暂无”等占位词到正式简历正文中

## 六、优化说明要求

你还需要输出 JSON 数组，记录关键调整。每个对象必须包含以下字段：

- category: 调整类别，只能是以下枚举之一：
  - "模块重组"
  - "内容改写"
  - "关键词对齐"
  - "信息删减"
  - "新增整合"
  - "风险提示"
- point: 一句话说明本次调整
- before: 调整前摘要；如果没有对应原文，填空字符串 ""
- after: 调整后摘要；如果只是风险提示，也可填空字符串 ""
- reason: 调整原因，说明与目标岗位的关系
- confidence: 只能是 "high"、"medium"、"low"
- needs_confirmation: 只能是 true 或 false

其中：
- 如果调整完全基于原文显性信息，confidence 设为 "high"
- 如果属于保守概括或结构性提炼，设为 "medium"
- 如果信息不足、存在解释空间、建议用户确认，设为 "low"，并将 needs_confirmation 设为 true

## 七、输出格式要求

严格按以下格式输出，不能添加任何额外说明、开场白、结尾、代码围栏：

===RESUME_START===
[优化后的简历 Markdown]
===RESUME_END===

===NOTES_START===
[合法 JSON 数组]
===NOTES_END===

## 八、最终自检

输出前请检查：
1. 是否存在编造事实
2. 是否把不确定内容写成了确定事实
3. JSON 是否合法可解析
4. Markdown 简历是否结构清晰
5. 输出中是否包含了分隔符之外的多余内容
```

#### User Prompt 模板

```
## 原始简历

{resume_text}

## 目标岗位

{target_position}

## 补充信息

{additional_info}
如果未提供，请按“无”理解。

## 额外要求
请优先保留真实信息，避免任何编造。
如果某些内容信息不足，请保守表达，并在优化说明中标记需要确认的点。
请直接按系统要求输出，不要添加额外说明。
```

### 5.3 Prompt 设计要点说明

| 设计决策 | 原因 |
|---------|------|
| 结构化分隔标记 `===XXX_START/END===` | 方便前端精确解析，不会与简历内容冲突 |
| 优化说明用 JSON 格式 | 前端可直接渲染为结构化卡片列表 |
| 强调"不可捏造" | 防止 AI 幻觉编造不存在的经历或数据 |
| 风险分级与确认标记 | 便于前端展示“需人工确认”的改写点 |
| 缺失信息保守处理 | 简历信息不完整时仍能稳定输出，不因缺字段而幻觉补全 |
| 单次请求内部分步思考 | 保留单次调用的简单性，同时提升长文本场景下的稳定性 |
| 单次请求而非多步 | 简历优化是单一、完整的任务，拆分反而降低连贯性 |

### 5.4 模型参数建议

| 参数 | 值 | 说明 |
|------|-----|------|
| model | `gpt-4o` | 优化质量优先，成本可接受（自用场景量不大） |
| temperature | `0.3` | 较低温度，保证输出稳定、专业，不过度发散 |
| max_tokens | `4096` | 简历 + 优化说明通常在 3000 token 以内 |

## 六、技术架构概览

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   前端 SPA   │ ───▶ │   后端 API   │ ───▶ │  OpenAI API │
│  (React)    │ ◀─── │  (Node.js)  │ ◀─── │  (GPT-4o)   │
└─────────────┘      └──────┬──────┘      └─────────────┘
                            │
                     ┌──────┴──────┐
                     │  文件解析    │
                     │ PDF → 文本  │
                     │ Word → 文本 │
                     └─────────────┘
```

- **前端**：React + TailwindCSS，单页应用
- **后端**：Node.js + Express，提供两个接口：
  - `POST /api/parse-file` — 解析上传的 PDF/Word 文件为文本
  - `POST /api/optimize-resume` — 调用 LLM 生成优化简历
- **无数据库**：不持久化存储，所有数据在会话内完成
